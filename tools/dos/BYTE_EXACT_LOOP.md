# Byte-exact build loop - Windows 1.03

This directory contains the infrastructure to **rebuild Windows 1.03 binaries
from source using the original 1985-1986 Microsoft toolchain**.

## What this enables

Take a function we've recovered (e.g. `GETVERSION` from `KERNEL.EXE`) and ask:

> If we lift this function from .asm back to a C source file and compile it
> with the **same** vintage compiler Microsoft used in 1985, do we produce
> the **exact same bytes** that ship in the retail `KERNEL.EXE`?

If the answer is yes, that single function is *byte-exact verified*: it is
guaranteed semantically faithful. This is the gold standard from the SM64
decomp / Pokemon decomp / etc. communities.

## Toolchain identification

Per @starfrost013 (2026-05-25 email): Windows 1.03 was built with
**Microsoft C 4.0 (1986)** and **MASM 4.0 (October 1985)**. The `cMacros`
PASCAL-flavored assembly seen throughout `src/*/seg*.asm` is generated by
that exact MSC 4.0 → MASM 4.0 chain.

The retail `LINK4.EXE` (segmented-executable linker) shipped on the
**Windows 1.01 ISV Toolkit** disk 8 supplement, version **5.00.12**, is
specifically designed to consume MSC 4.0 `.OBJ` files and produce `NE`
executables for Windows 1.x.

```
tools/dos/
  Masm40.img              MASM 4.0 floppy image (Oct 1985, 360 KB)
  disk01..disk08.img      MSC 4.0 floppy images (1986, 8 x 360 KB)
  extracted/<diskname>/   Files copied off each floppy
  combined/               All toolchain binaries flat in one dir
  work/                   Sandbox for build experiments
```

## Source of the toolchain (legal abandonware)

- MSC 4.0: <https://archive.org/details/microsoft-c-compiler-4.0-5.25-360k.-7z>
- MASM 4.0: <https://archive.org/details/microsoft-macro-assembler-4.0-5.25.-7z>

These are 40-year-old Microsoft binaries archived by the Internet Archive.
Microsoft has no commercial interest in MSC 4.0 anymore; modern Visual Studio
descends from MSC 6+ which is a different codebase.

These functions come from three different .EXE binaries (KERNEL, USER,
SYSTEM), include real DOS interrupts, FAR string operations, segment-
override trickery, and multi-branch logic. They are all reproducible on
demand by running:

```bash
python bootstrap/analyze/pass23_find_ministubs.py   # picks candidates
python bootstrap/analyze/pass24_batch_masm_verify.py # MASMs them all
```

The Python harness (Pass 24) automates the dirty work: pretty-printer
syntax cleanup, MASM 4.0 dialect coercion, dangling-label patching via
Capstone disassembly, and a single-spawn DOSBox-X batch run that
assembles all candidates in ~12 seconds total.

## First byte-exact match: GETVERSION

Windows 1.03 `KERNEL.EXE` exposes ordinal 3 = `GETVERSION`. The NE entry
table places it at segment 1, offset `0x023C`. The 4 raw bytes there are:

```
B8 01 03    mov ax, 0x0301    ; major=1, minor=3 (Windows 1.03)
CB          retf              ; FAR return
```

The lift back to MASM 4.0 source:

```asm
_TEXT   SEGMENT BYTE PUBLIC 'CODE'
        ASSUME  CS:_TEXT
PUBLIC  GETVERSION
GETVERSION PROC FAR
        mov     ax, 0301h
        ret
GETVERSION ENDP
_TEXT   ENDS
        END
```

Built with `masm getver,getver,getver,nul;` on disk 6 of the original
1985 Microsoft Macro Assembler 4.0 floppy, the produced `GETVER.OBJ`
contains the bytes `B8 01 03 CB` at OBJ offset `0x26`. These four bytes
match the retail `KERNEL.EXE` byte-for-byte. **First function whose
behavior we can claim to have not just understood but fully reproduced
from source with the original toolchain.**

The `cMacros` `cProc`-style prologues we see throughout the rest of the
codebase mean that more substantial C functions will introduce
`__chkstk`, the FAR-PASCAL inc-bp/push-bp/mov-bp,sp/push-ds/mov-ds,ax
preamble, and the `sub bp,2 / mov sp,bp / pop ds / pop bp / dec bp / retf`
epilogue. We confirmed all of these are emitted by `cl /AS /Gw` matching
exactly the patterns we see in `src/<module>/seg*.asm`.

## Critical lessons learned (the hard way)

1. **DOSBox-X internal `-time-limit N` is mandatory.** External `timeout` from
   Linux does **not** kill DOSBox-X reliably when COMMAND.COM is waiting on
   keyboard input - the process becomes a zombie and hangs WSL itself.
   Always pass `-time-limit 25` (or whatever budget) to DOSBox-X directly.

2. **MSC 4.0 is K&R, not ANSI.** `int f(void)` produces
   `error 55: expected formal parameter list, not a type list`. Use
   `int f()` and old-style parameter declarations:
   ```c
   int sum(a, b) int a; int b; { return a + b; }
   ```

3. **DOS `COMMAND.COM` does not support `2>&1`** redirection. Use only
   `cmd > log.txt` for stdout. Errors go to the screen and are lost unless
   you check the produced file's existence as a success indicator.

4. **LINK 3.51 is interactive.** It prompts for 4 things (Object, Run File,
   Map, Libraries). The trick to avoid hangs is `cmd < nul` - feeding EOF
   to stdin makes it accept defaults and skip.

5. **`cl /AS tiny.c` (no `/c`) auto-runs LINK** and pulls in CRT0 / SLIBC
   automatically. This is way easier than calling LINK manually.
